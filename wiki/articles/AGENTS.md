# Articles — 執筆ガイドライン

article を書く際の粒度・構成・わかりやすさの指針．

## 記事の粒度

- **1概念 1ファイル**．独立して理解できる単位で切る．複数の概念を組み合わせて「全体でどう機能するか」を説明することはしない．それは別レイヤーである synthesis の役割である．

## 記事の構成パターン

### 1. 導入（冒頭1-2段落）

- **問題起点**で書く．「〇〇が増えると△△がわからなくなる．この概念はその問題を解決する」
- **存在意義起点**で書く．「これがあると〇〇ができるようになる．」
- 用語の定義ではなく，**なぜこの概念が必要か**から入る．

### 2. 全体像

- テーブルや ASCII ツリーで構造を一望できるようにする．
- 例: テーブル，データモデルの階層図．

### 3. 具体例で理解する（Option）

- **シナリオベース**のステップで説明する．実際の操作に沿って概念を導入する．
- 各ステップは `**Step N — 動詞で始まるタイトル**` + 説明文の形式．
- コード例や設定例を含めて具体性を持たせる．

### 4. 設計思想（なぜこの構造なのか）

- 仕様の列挙ではなく，**設計上の判断とその理由**を説明する．
- 例: 「付与と強制の分離」「システム vs カスタムの二層構造」．

### 5. リファレンス（記事末尾）

- ルックアップ用のテーブルを配置．読み返すときに素早く参照できるようにする．
- 構成要素一覧，制約，分類表など．

## 文体・表記

- 日本語で記述．句読点はコンマ(，)とピリオド(．)を使用．
- 関連概念は `[[Concept_Name]]` で参照（Obsidian wikilink）．
- 見出しは内容に即した具体的な名前にする（「概要」「詳細」のような汎用名は避ける）．
- 太字は強調したいキーワードに限定して使う．

## やること・やらないこと

| やること | やらないこと |
| --- | --- |
| 問題起点で「なぜ必要か」から書く | 用語辞典のように定義を列挙する |
| シナリオで具体的に理解させる | 抽象的な説明だけで終わる |
| 設計の「なぜ」を説明する | 仕様を羅列する |
| リファレンスを末尾にまとめる | 本文中に参照用テーブルを散在させる |
| wikilink で概念間をつなぐ | 記事内で他の概念を詳しく再説明する |
